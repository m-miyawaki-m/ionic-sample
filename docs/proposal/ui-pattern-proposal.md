# 画面UIパターン提案書

倉庫在庫管理アプリの画面デザインについて、各画面で複数のパターンをご用意しました。
実際にアプリ上で操作しながら、各パターンを比較いただけます。
本資料では、それぞれのパターンのメリット・デメリットと推奨案をまとめています。

## 1. ホーム画面（メニュー一覧）

アプリ起動後、最初に表示されるメニュー画面のレイアウトです。

### パターン A: リスト表示

```
┌──────────────────────────┐
│ 倉庫管理            [≡]  │
├──────────────────────────┤
│ 📥 入荷検品              │
│    商品スキャン→入荷登録  │
├──────────────────────────┤
│ 📤 出荷検品              │
│    出荷指示照合→結果送信  │
├──────────────────────────┤
│ 📋 棚卸し                │
│    ロケーション→数量確認  │
├──────────────────────────┤
│ 🔍 在庫照会              │
│    品目スキャン→在庫表示  │
├──────────────────────────┤
│ ↔ ロケーション移動       │
│    移動元→移動先→品目     │
└──────────────────────────┘
```

| メリット | デメリット |
|---------|----------|
| 各メニューに説明文を表示でき、初めての作業者でも迷わない | 画面スクロールが必要になる場合がある |
| 1列なので目線の移動が少ない | メニュー数が少ないと余白が目立つ |
| 一般的なアプリに多い形式で違和感が少ない | |

### パターン B: グリッド（カード）表示

```
┌──────────────────────────┐
│ 倉庫管理            [≡]  │
├──────────────────────────┤
│ ┌──────────┐┌──────────┐ │

│ │   📋     ││   🔍     │ │
│ │ 棚卸し   ││ 在庫照会  │ │
│ └──────────┘└──────────┘ │
│ ┌──────────┐             │
│ │   ↔      │             │
│ │ ロケ移動  │             │
│ └──────────┘             │
└──────────────────────────┘
```

| メリット | デメリット |
|---------|----------|
| アイコンが大きく表示され、視覚的にわかりやすい | 説明文を入れるスペースが少ない |
| スクロール不要で全メニューが一覧できる | メニュー数が奇数だと1つ余る |
| スマホアプリらしい見た目 | 文字が小さくなりがち |

### パターン C: 大ボタン表示

```
┌──────────────────────────┐
│ 倉庫管理            [≡]  │
├──────────────────────────┤
│                          │

│ │  📤  出荷検品       │   │

│ │  📋  棚卸し         │   │

│                          │
│ ┌────────────────────┐   │
│ │  🔍  在庫照会       │   │
│ └────────────────────┘   │

│ │  ↔  ロケーション移動│   │
│ └────────────────────┘   │

```

| メリット | デメリット |
|---------|----------|
| **ボタンが大きく、手袋をしたままでも確実に押せる** | スクロールが必要 |
| 押し間違いが最も少ない | 説明文を表示するスペースがない |
| 文字が大きく、離れた距離からでも読める | 画面の情報量は最も少ない |

### 推奨

| 利用環境 | 推奨パターン | 理由 |
|---------|------------|------|
| **手袋着用の倉庫作業** | **C（大ボタン）** | 押し間違い防止。作業手袋でもタップしやすい |
| オフィスでの管理作業 | B（グリッド） | 全メニューが一覧でき、見た目も良い |
| 初めて使う作業者が多い | A（リスト） | 説明文があり迷わない |

## 2. 業務画面（入力フォーム）

入荷検品・出荷検品・棚卸し・ロケーション移動の各業務画面で共通のフォームレイアウトです。

### パターン A: 縦並びフォーム

```
┌──────────────────────────┐
│ ← 入荷検品          [≡]  │
├──────────────────────────┤
│ [スキャナ: 接続中]        │
│                          │
│ ロケーション              │
│ [A-01-03            ] [📷]│
│                          │
│ 品目コード                │
│ [ITEM-001           ] [📷]│
│                          │
│ 数量                     │
│ [    1              ]    │
│                          │
│ ロット番号                │
│ [LOT-2026-001      ] [📷]│
│                          │
│ [        登録        ]   │
└──────────────────────────┘
```

| メリット | デメリット |
|---------|----------|
| **全項目が一覧でき、入力状況を把握しやすい** | 項目が多いとスクロールが必要 |
| 一般的なフォームと同じ形式で違和感がない | スキャン項目と手入力項目の区別がつきにくい |
| 熟練作業者は最も速く操作できる | |

### パターン B: グループ分け（カード）表示

```
┌──────────────────────────┐
│ ← 入荷検品          [≡]  │
├──────────────────────────┤
│ [スキャナ: 接続中]        │
│                          │
│ ┌─ スキャン項目 ────────┐ │
│ │ ロケーション           │ │
│ │ [A-01-03         ][📷]│ │
│ │ 品目コード             │ │
│ │ [ITEM-001        ][📷]│ │
│ │ ロット番号             │ │
│ │ [LOT-2026-001   ][📷]│ │
│ └────────────────────────┘│
│ ┌─ 入力項目 ────────────┐ │
│ │ 数量                   │ │
│ │ [    1              ] │ │
│ └────────────────────────┘│
│                          │
│ [        登録        ]   │
└──────────────────────────┘
```

| メリット | デメリット |
|---------|----------|
| **スキャンする項目と手入力する項目が一目でわかる** | 画面の占有面積が大きい |
| 作業の種類（スキャン/入力）が視覚的に分離される | 縦並びより情報量が増える |
| 初めての作業者への研修で説明しやすい | |

### パターン C: ステッパー（1項目ずつ表示）

```
┌──────────────────────────┐
│ ← 入荷検品          [≡]  │
├──────────────────────────┤
│ [スキャナ: 接続中]        │
│                          │
│ ステップ 1/4: ロケーション│
│ ■■■■□□□□□□□□            │
│                          │
│                          │
│ ロケーション              │
│ [                   ] [📷]│
│                          │
│                          │
│                          │
│ [  戻る  ] [   次へ   ]  │
└──────────────────────────┘
```

| メリット | デメリット |
|---------|----------|
| **1項目に集中でき、入力ミスが減る** | 全体の入力状況が見えない |
| 表示が大きく見やすい。遠くからでも確認しやすい | 項目数が多いとステップ数が増えて手間 |
| **進捗バーで「あと何項目か」がわかる** | 前の項目を修正するには戻る操作が必要 |
| 新人研修の初期段階で効果的 | 熟練者には遅く感じる |

### 推奨

| 利用者 | 推奨パターン | 理由 |
|--------|------------|------|
| **新人・研修中** | **C（ステッパー）** | 1項目ずつ確認でき、ミスを防げる |
| **日常業務（標準）** | **A（縦並び）** | 最も速く操作でき、慣れている形式 |
| 作業手順を明確にしたい | B（グループ分け） | スキャン/手入力の区別が明確 |

> **運用提案**: 新人にはCで研修し、慣れたらAに切り替える、という段階的な運用が可能です。
> 切替は右上のメニューからいつでも変更でき、設定は端末に保存されます。

## 3. メニュー（右上ハンバーガー）

各画面の右上にある「≡」ボタンを押した時の表示方式です。
ここには「QRコード読み取り」「バーコード読み取り」等のスキャンモード切替や、ダークモード切替が入ります。

### パターン A: ポップオーバー（吹き出し）

```
                      [≡] ← 押す
                    ┌──────────────┐
                    │QRコード読み取り│
                    │バーコード読取り│
                    │──────────────│
                    │ダークモード   │
                    └──────────────┘
```

| メリット | デメリット |
|---------|----------|
| **ボタンの近くに表示され、直感的** | 項目数が多いと画面を覆う |
| 軽い操作感。開閉が速い | 画面外をタップで閉じる操作が必要 |
| 表示領域が小さく、画面を遮らない | |

### パターン B: アクションシート（下からスライド）

```
┌──────────────────────────┐
│ ← 入荷検品          [≡]  │← 押す
├──────────────────────────┤
│                          │
│  （画面内容が暗くなる）   │
│                          │
├──────────────────────────┤
│ スキャンモード選択        │
│                          │
│ QRコード読み取り          │
│ バーコード読み取り        │
│ ダークモード             │
│                          │
│ [キャンセル]              │
└──────────────────────────┘
```

| メリット | デメリット |
|---------|----------|
| **画面下部に表示され、片手で操作しやすい** | 画面全体が暗くなり、背景が見えにくい |
| 選択肢が大きく表示される | 表示・非表示のアニメーションに時間がかかる |
| 「キャンセル」ボタンが明確 | ポップオーバーより操作が1ステップ多い |

### パターン C: サイドメニュー（横からスライド）

```
┌──────────────────────────┐
│ ← 入荷検品          [≡]  │← 押す
├──────────────────────────┤
│              ┌───────────┤
│              │ メニュー   │
│              │           │
│              │ QRコード   │
│              │ バーコード │
│              │ ダークモード│
│              │           │
│              │ [閉じる]   │
│              └───────────┤
└──────────────────────────┘
```

| メリット | デメリット |
|---------|----------|
| 項目数が多くても対応できる | メニューの開閉で画面が大きく変わる |
| 設定画面のような見た目で、設定項目向き | 単純な選択操作にはオーバー |
| スワイプで開閉できる | 他の操作と干渉する場合がある |

### 推奨

| 用途 | 推奨パターン | 理由 |
|------|------------|------|
| **スキャンモード切替（3〜5項目）** | **A（ポップオーバー）** | 項目数が少ないので、軽い操作で十分 |
| 設定項目が10項目以上に増える場合 | C（サイドメニュー） | 多くの項目をスクロールで表示 |
| 片手操作を重視する場合 | B（アクションシート） | 画面下部で親指が届く |

## 4. ローディング（通信中の表示）

サーバーとの通信中（登録・検索等）に、処理中であることをユーザーに伝える方法です。

### パターン A: 全画面オーバーレイ

```
┌──────────────────────────┐
│ ← 入荷検品          [≡]  │
├──────────────────────────┤
│▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒│
│▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒│
│▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒│
│▒▒▒▒▒▒▒ 🔄 ▒▒▒▒▒▒▒▒▒▒▒▒▒│
│▒▒▒▒▒▒ 登録中... ▒▒▒▒▒▒▒▒│
│▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒│
│▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒│
│▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒│
└──────────────────────────┘
  ▒ = 半透明の暗い背景
  画面操作は一切できない
```

| メリット | デメリット |
|---------|----------|
| **処理中であることが明確にわかる** | 画面操作が完全にブロックされる |
| **二重送信（登録ボタンの連打）を防止できる** | 通信が遅い場合、ユーザーが不安になる |
| 「今は待っていてください」が伝わる | キャンセル操作ができない |

### パターン B: ボタン内スピナー

```
┌──────────────────────────┐
│ ← 入荷検品          [≡]  │
├──────────────────────────┤
│                          │
│ ロケーション              │
│ [A-01-03            ] [📷]│
│ 品目コード                │
│ [ITEM-001           ] [📷]│
│ 数量                     │
│ [    1              ]    │
│                          │
│ [ 🔄 送信中...       ]   │ ← ボタンだけがグルグル
│                          │  他の項目は操作可能
└──────────────────────────┘
```

| メリット | デメリット |
|---------|----------|
| 画面の他の部分は引き続き確認できる | **登録ボタンの連打を防ぐ追加対策が必要** |
| 軽い印象で、ストレスが少ない | 処理中であることが目立ちにくい |
| 入力値の確認や修正が通信中でもできる | |

### 推奨

| 操作 | 推奨パターン | 理由 |
|------|------------|------|
| **データ登録・更新（入荷、出荷、棚卸し等）** | **A（全画面）** | **二重送信防止が最も重要。** 同じデータが2回登録されると業務に影響する |
| データ検索・照会 | B（ボタン内） | 検索中に入力値を変更したいケースがある |

> **重要**: データ登録系の操作では、二重送信（同じ入荷データが2件登録される等）を
> 確実に防ぐ必要があるため、全画面オーバーレイを推奨します。

## 5. スキャンフィードバック（読み取り時の通知）

SP2スキャナでバーコード/QRコードを読み取った際に、成功したことを作業者に伝える方法です。
倉庫環境では**画面を見ていないタイミングでスキャンする**場面が多いため、
視覚以外のフィードバックが重要です。

### パターン A: フィードバックなし

スキャン結果は画面の入力欄に入るだけ。それ以外の通知なし。

| メリット | デメリット |
|---------|----------|
| 実装がシンプル | **スキャン成功に気づかない可能性がある** |
| 連続スキャン時に邪魔にならない | 失敗時にもわからない |

### パターン B: バイブレーション（振動）のみ

スキャン成功時に端末が短く振動する（ブッ）。

| メリット | デメリット |
|---------|----------|
| **画面を見ていなくてもスキャン成功がわかる** | 何を読み取ったかは画面を見ないとわからない |
| **手袋をしていても振動は伝わる** | 静かな環境では他の人に気づかれない |
| バッテリーへの影響はほぼなし | 端末をテーブルに置いていると振動に気づきにくい |

### パターン C: トースト通知（画面表示）のみ

画面下部に「読み取り: ITEM-001」と1.5秒間表示。

```
┌──────────────────────────┐
│                          │
│  （通常の画面）           │
│                          │
│ ┌──────────────────────┐ │
│ │ 読み取り: ITEM-001    │ │ ← 1.5秒で消える
│ └──────────────────────┘ │
└──────────────────────────┘
```

| メリット | デメリット |
|---------|----------|
| 読み取った値がすぐ確認できる | **画面を見ていないと気づかない** |
| 誤読み取りをすぐ発見できる | 手元を見ずにスキャンする作業には不向き |

### パターン D: バイブレーション + トースト通知

振動（ブッ）＋画面表示「読み取り: ITEM-001」の両方。

| メリット | デメリット |
|---------|----------|
| **画面を見ていなくても振動で気づく** | 特になし |
| **画面を見れば読み取り値も確認できる** | |
| 成功・失敗を触覚と視覚の両方で確認 | |

### 推奨

| 環境 | 推奨パターン | 理由 |
|------|------------|------|
| **倉庫作業（標準）** | **D（バイブ＋トースト）** | 手袋をしていても振動で成功がわかり、画面で値も確認できる |
| 静かなオフィス | C（トースト） | 振動音が気になる場合 |
| 連続高速スキャン | B（バイブ） | トースト表示が邪魔になる場合 |

> **追加提案**: エラー時（読み取り失敗）には、成功時より**長い振動 + 赤いトースト**で
> 明確に区別できるようにすることを推奨します。

## 6. 判定シート

以下の表に、各カテゴリの採用パターンをご記入ください。

| # | カテゴリ | パターン候補 | 推奨 | **採用** |
|---|---------|------------|------|---------|
| 1 | ホーム画面 | A:リスト / B:グリッド / C:大ボタン | 環境依存 | |
| 2 | 業務画面フォーム | A:縦並び / B:グループ / C:ステッパー | A（日常）C（研修） | |
| 3 | メニュー表示 | A:ポップオーバー / B:アクションシート / C:サイドメニュー | A | |
| 4-a | ローディング（登録系） | A:全画面 / B:ボタン内 | A | |
| 4-b | ローディング（検索系） | A:全画面 / B:ボタン内 | B | |
| 5 | スキャンフィードバック | A:なし / B:バイブ / C:トースト / D:バイブ+トースト | D | |

### 補足事項

- **複数パターンの併用が可能です。** 例えば、ホーム画面はC（大ボタン）を標準にしつつ、
  右上メニューからA（リスト）やB（グリッド）に切り替えられる仕様にできます。
- **端末ごとに設定を変えることも可能です。** 例えば、新人用端末はステッパー表示、
  熟練者用端末は縦並び表示、のように設定を分けられます。
- **設定は端末に保存されます。** アプリを再起動しても、選択したパターンが維持されます。
- **全パターンはデモアプリで実際に操作して確認いただけます。**
  右上の「≡」メニューからパターンを切り替えてお試しください。

### 確認方法

1. デモアプリを端末またはブラウザで開く
2. 各画面の右上「≡」メニューを開く
3. 表示パターン（A/B/C）を選んで切り替える
4. 操作感を確認し、上記の判定シートに記入する

---

## 参考: パターン採用例

### 例1: 標準的な倉庫環境

| カテゴリ | 採用 | 理由 |
|---------|------|------|
| ホーム画面 | C（大ボタン） | 手袋着用のため大きいボタンが必要 |
| 業務画面 | A（縦並び） | 作業者は経験者が多い |
| メニュー | A（ポップオーバー） | 項目数が少なく、軽い操作で十分 |
| ローディング（登録） | A（全画面） | 二重登録防止が最優先 |
| ローディング（検索） | B（ボタン内） | 検索中に入力修正したい場合がある |
| フィードバック | D（バイブ+トースト） | 手袋越しでも振動で気づく |

### 例2: 新人研修環境

| カテゴリ | 採用 | 理由 |
|---------|------|------|
| ホーム画面 | A（リスト） | 説明文で各機能がわかる |
| 業務画面 | C（ステッパー） | 1項目ずつ確認でき、ミスを防げる |
| メニュー | A（ポップオーバー） | 研修中はメニュー操作を減らしたい |
| ローディング（登録） | A（全画面） | 二重登録防止 |
| ローディング（検索） | A（全画面） | 研修中は待つことに集中させる |
| フィードバック | D（バイブ+トースト） | 成功/失敗が明確にわかる |

### 例3: オフィス環境（管理者用）

| カテゴリ | 採用 | 理由 |
|---------|------|------|
| ホーム画面 | B（グリッド） | 見た目が良く、全メニュー一覧 |
| 業務画面 | B（グループ分け） | スキャン/入力の区別が明確 |
| メニュー | A（ポップオーバー） | 標準的で違和感がない |
| ローディング（登録） | A（全画面） | 二重登録防止 |
| ローディング（検索） | B（ボタン内） | 軽い操作感 |
| フィードバック | C（トーストのみ） | オフィスで振動は不要 |
