# 倉庫在庫管理アプリ 設計書

## 概要

自衛隊向け受託開発の倉庫在庫管理アプリ。Android端末（Xnavis）上で動作し、SP2スキャナでバーコード/QRコードを読み取り、外部システムとJSON HTTP通信でデータ連携を行う。

## 技術スタック

| レイヤー | 技術 |
|---|---|
| UI フレームワーク | Ionic Vue (TypeScript) |
| ネイティブランタイム | Capacitor |
| ルーティング | Vue Router (Ionic統合) |
| 状態管理 | Pinia（必要に応じて） |
| HTTP通信 | fetch または Axios（JSON形式） |
| スキャナ連携 | Capacitor Local Plugin → SP2 AAR SDK |
| ビルド | Capacitor CLI + Android Studio |

## SP2プラグイン設計

### プラグイン方式

**Capacitor Local Plugin（プロジェクト内直接実装）** を採用する。

独立したnpmパッケージとしてのカスタムプラグイン（アプローチA）も検討したが、以下の理由でLocal Pluginとした:

- Ionic/Capacitorが初めてのため、パッケージ構造の学習負担を減らす
- 単一の受託プロジェクト向けであり、再利用性の優先度が低い
- プロジェクト内で完結するため、ビルド構成がシンプル
- 後からカスタムプラグイン（別パッケージ）に切り出すことは容易

> **比較詳細**: アプローチA（カスタムプラグイン）とアプローチB（Local Plugin）では、Javaで書くブリッジコードは同一。違いは配置場所のみ。Aは `sp2-scanner-plugin/` として別パッケージに分離し npm install で取り込む。Bは `android/app/` 内に直接配置する。

### ファイル構成

```
android/app/
├── libs/
│   └── sp2-scanner.aar            ← SP2 SDK
├── build.gradle                   ← libs依存を追加
└── src/main/java/
    └── com/example/sp2/
        ├── SP2Plugin.java         ← Capacitorプラグイン本体
        └── SP2Handler.java        ← AAR SDK操作のヘルパー（任意）
```

### プラグインAPI

| メソッド | 役割 | 戻り値 |
|---|---|---|
| `initialize()` | AAR SDKの初期化・スキャナ接続 | 成功/失敗 |
| `startScan()` | スキャン開始（1回読み取り） | - |
| `stopScan()` | スキャン停止 | - |
| `getStatus()` | スキャナの接続状態を取得 | connected / disconnected |
| `destroy()` | SDK解放・切断 | - |

スキャン結果はCapacitorのイベントリスナーで通知する。

### Vue側の利用イメージ

```typescript
const { startScan, stopScan, onScanResult } = useSP2Scanner()

onScanResult((result) => {
  // result.value  → "4901234567890"
  // result.format → "CODE128" / "QR_CODE" など
  // → 現在フォーカス中の入力欄に値をセット
})
```

## スキャン入力フロー

画面に複数の入力欄があり、フォーカスしている欄にスキャン結果を順番に入力する方式。

```
1. 入力欄にフォーカス
2. アプリが startScan() を呼ぶ（またはユーザー操作でトリガー）
3. SP2が1件読み取る → イベントで結果が返る
4. 現在フォーカス中の欄に値がセットされる
5. 次の欄にフォーカスを移す
6. 繰り返し
```

## 外部API連携

- 通信形式: JSON over HTTP (REST想定)
- API仕様は後日確定。インターフェースだけ先に定義し、実装は差し替え可能にする
- 認証方式は未定（端末認証の可能性あり）

## データ永続化

- アプリ内にローカルDBは持たない
- 外部システムがデータのマスター
- サーバーへのリアルタイム接続が前提（オフライン対応なし）

## 対象端末

- OS: Android
- 端末: Xnavis
- スキャナ: SP2（AAR SDK提供、接続方式はSDKが抽象化）

## 検討の記録

### プラグイン方式の比較（2026-04-01）

SP2 AAR SDKをIonic/Capacitorから利用する方式として3案を検討した。

#### アプローチA: Capacitor カスタムプラグイン（別パッケージ）

- SP2プラグインを独立したnpmパッケージとして作成
- `npm install` でアプリに取り込む
- メリット: 他プロジェクトでの再利用が容易、プラグインの責務が明確に分離
- デメリット: パッケージ構造（TypeScript定義 + Java実装 + package.json + ビルド設定）の理解が追加で必要

#### アプローチB: Capacitor Local Plugin（プロジェクト内直接実装）★ 採用

- `android/app/` 内にJavaコードを直接配置
- メリット: 最も手軽に始められる、プロジェクト内で完結、ビルド構成がシンプル
- デメリット: 他プロジェクトでの再利用がしにくい
- 採用理由: 初めてのIonic開発での学習負担軽減、単一プロジェクト向け、後からAに切り出し可能

#### アプローチC: WebView + Intent/BroadcastReceiver 経由

- SP2のAARを別のネイティブサービスとして実装し、Intent/BroadcastReceiverで通信
- メリット: スキャナのライフサイクルをアプリと分離、バックグラウンドスキャン対応しやすい
- デメリット: 実装が複雑、過剰設計、Intentベースの通信はデバッグが難しい
- 不採用理由: 今回の要件に対して過剰

#### 結論

AとBではJavaブリッジコードは同一で、違いはファイル配置のみ。学習コストとプロジェクト規模を考慮しBを採用。Cは過剰設計のため不採用。
