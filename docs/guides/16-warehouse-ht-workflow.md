# 倉庫HT（ハンディターミナル）業務フロー

倉庫現場でハンディターミナル（HT）を利用する際の典型的な業務フローを整理する。
本アプリの画面設計はこのフローに基づいている。

## 1. 基本原則

倉庫HTの操作は **「スキャンできるものはスキャン、数量だけ手入力、最後に確定」** が基本。

| 原則 | 理由 |
|------|------|
| スキャンで入力できるものは手入力しない | 入力ミス防止、作業速度向上 |
| 数量は人が数えて手入力する | バーコードに数量情報はない |
| 確定は必ずボタン操作 | 誤登録防止のため明示的な操作が必要 |
| 1トランザクションは短く完結させる | 作業中断・端末障害時のデータロスを最小化 |

---

## 2. 共通フロー

すべての業務は以下の共通フローに従う。

```
メニュー画面
  └→ 業務画面を選択
       └→ 伝票/指示を特定（スキャン or リスト選択）
            └→ ロケーションをスキャン
                 └→ 商品バーコードをスキャン
                      └→ 数量を手入力（必要な場合）
                           └→ 確定ボタン押下
                                └→ サーバーに送信
                                     └→ 次の商品 or 完了
```

### ステップごとの詳細

| ステップ | 操作 | 補足 |
|---------|------|------|
| 業務選択 | メニュー画面からタップ | 作業者の権限で表示を制御する場合もある |
| 伝票/指示の特定 | 紙伝票のバーコードスキャン or 画面リストから選択 | 予め割り当てられた指示が表示されるケースもある |
| ロケーション入力 | 棚に貼付されたバーコードをスキャン | 棚番は `A-03-02` のような形式が多い |
| 商品入力 | 商品に印刷されたバーコードをスキャン | JANコード、ITFコード、社内コードなど |
| 数量入力 | テンキーで手入力 | +1カウントアップ方式の場合もある |
| 確定 | ボタン押下 | サーバーと通信、結果をフィードバック |

## 3. 業務別フロー

### 3.1 入荷検品

入荷した商品が伝票通りか確認し、在庫に登録する。

```
入荷検品画面を開く
  → 入荷伝票番号をスキャン（または手入力）
  → 伝票内容が画面に表示される
  ─── 以下、商品ごとに繰り返し ───
  → ロケーション（格納先）をスキャン
  → 商品バーコードをスキャン
  → 数量を手入力
  → 「登録」ボタン → サーバー送信
  ─── 繰り返し終了 ───
  → 全品登録完了 → 伝票完了処理
```

**対応画面:** `ReceivingPage.vue`

### 3.2 出荷（概要）

出荷業務は複数のステップで構成される。ここでは全体像を示し、詳細は [4. 出荷業務の詳細フロー](#4-出荷業務の詳細フロー) で解説する。

```
出荷指示受信 → 現品登録（ピッキング+検品） → 発送準備登録 → 発送 → 発送完了登録
```

**対応画面:** `ShippingPage.vue`（現品登録・検品部分）

### 3.3 棚卸し

実際の在庫数をカウントし、システム在庫との差異を確認する。

```
棚卸し画面を開く
  → ロケーションをスキャン
  ─── 以下、商品ごとに繰り返し ───
  → 商品バーコードをスキャン
  → 実数量を手入力
  → 「登録」ボタン
  ─── 繰り返し終了 ───
  → ロケーション完了 → 次のロケーションへ
```

**ポイント:** 伝票がない。ロケーション単位で棚にある商品をすべてカウントする。システム在庫との差異は後工程で確認。

**対応画面:** `StocktakingPage.vue`

### 3.4 在庫照会

特定の商品やロケーションの在庫情報を確認する（登録は行わない）。

```
在庫照会画面を開く
  → 商品バーコードをスキャン（またはロケーションをスキャン）
  → 在庫情報が表示される（ロケーション、数量、ロット等）
```

**ポイント:** 参照のみ。データ登録がないため確定ボタンは不要。

**対応画面:** `InventoryPage.vue`

### 3.5 ロケーション移動

商品を別の棚に移す際に記録する。

```
ロケーション移動画面を開く
  → 移動元ロケーションをスキャン
  → 商品バーコードをスキャン
  → 数量を手入力
  → 移動先ロケーションをスキャン
  → 「登録」ボタン → サーバー送信
```

**ポイント:** 移動元と移動先の2つのロケーションが必要。

**対応画面:** `RelocationPage.vue`

## 4. 出荷業務の詳細フロー

出荷業務は以下の4ステップで進行する。

### 全体フロー

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  出荷指示   │ →  │  現品登録   │ →  │ 発送準備登録 │ →  │ 発送完了登録 │
│  受信       │    │（ピッキング │    │（梱包・荷合  │    │（積込・実績  │
│             │    │  ＋検品）   │    │  わせ）      │    │  送信）      │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
  ERPから降りる       HT: メイン作業      HT: 梱包確認       HT: 最終確認
```

### ステップ間のステータス遷移

| ステータス | 意味 | 次に進む条件 |
|-----------|------|-------------|
| 指示受信済み | ERPから出荷指示が降りた状態 | 作業者に割当 or 作業者が選択 |
| ピッキング中 | 棚から商品を取り出し中 | 全品スキャン完了 |
| 検品済み | 指示と現品が照合一致 | 検品確定ボタン押下 |
| 梱包済み | 箱詰め・送り状貼付完了 | 梱包確定ボタン押下 |
| 出荷済み | トラックに積込完了 | 発送完了登録 |

---

### 4.1 出荷指示受信

ERPや受注システムからWMSに出荷指示が連携される。

```
出荷指示一覧画面を開く
  → 自分に割り当てられた指示がリスト表示される
  → 指示を選択（またはバーコードスキャン）
  → 指示内容を確認（出荷先、品目一覧、数量、納期）
```

**HTでの操作:**
- 指示一覧から選択するだけの場合が多い
- 紙の指示書にバーコードが印刷されている場合はスキャンで選択
- 作業者への自動割当がある場合は、ログイン時に指示が表示される

---

### 4.2 現品登録（ピッキング + 検品）

出荷指示に対して実際の商品をスキャンし、照合する。出荷業務のメイン作業。

```
現品登録画面を開く
  → 出荷指示を選択（4.1で選択済みなら自動遷移）
  → 指示内容が表示される（品目・数量・ロケーション・出荷先）
  ─── 以下、商品ごとに繰り返し ───
  → ピッキング先ロケーションを表示（システムが指示）
  → ロケーションに移動し、ロケーションバーコードをスキャン
  → 商品バーコードをスキャン
  → 指示と照合
      ├─ 一致 → OK音 + 画面に✓表示
      └─ 不一致 → エラー音 + 警告「指示にない商品です」
  → 数量を確認（必要なら手入力）
  ─── 繰り返し終了 ───
  → 全品照合完了 → 「検品確定」ボタン
```

**ポイント:**
- 「照合」がメイン。スキャンした商品が指示にあるかどうかをリアルタイムで判定
- FIFO/FEFO対応の場合、システムがピッキングすべきロット・ロケーションを指定する
- 数量過不足がある場合は、理由コード（欠品・破損等）の入力を求めることがある

**シングルピッキング vs トータルピッキング:**

| 方式 | HT画面の違い |
|------|-------------|
| シングル（摘み取り） | 1出荷指示ずつ画面に表示。ピッキング→検品を連続で行う |
| トータル（種まき） | 複数指示をまとめて商品別にピッキング → 後から指示別に仕分けスキャン |

---

### 4.3 発送準備登録（梱包・荷合わせ）

検品済みの商品を梱包し、出荷できる状態にする。

```
発送準備画面を開く
  → 検品済みの出荷指示を選択
  ─── 以下、梱包単位ごとに繰り返し ───
  → 段ボール/コンテナのバーコードをスキャン（箱番号の紐付け）
  → 梱包する商品をスキャン（どの商品をどの箱に入れたか記録）
  → 数量を確認
  ─── 繰り返し終了 ───
  → 送り状番号をスキャン（配送業者の伝票番号と紐付け）
  → 「梱包確定」ボタン
```

**HTでの操作:**
- 梱包作業そのものはHTなしで行い、完了後にスキャンで紐付けだけ行うケースも多い
- 送り状の発行は別システム（配送業者API）の場合がある
- 荷合わせ（同一出荷先の複数箱をまとめる）はスキャンで箱番号を紐付け

**簡易パターン（小規模倉庫）:**
梱包を個別管理しない場合は、このステップを省略して検品完了→発送完了に進む。

---

### 4.4 発送完了登録

トラックへの積込が完了し、倉庫から商品が出たことを記録する。
このステップでシステム上の在庫が正式に引き落とされ、ERPに出荷実績が返される。

```
発送完了画面を開く
  → 出荷方法を選択
      ├─ A) 送り状スキャン方式（個別確認）
      └─ B) トラック単位一括方式

  ── A) 送り状スキャン方式 ──
  → トラック番号 or 配送業者を選択（手入力 or リスト選択）
  ─── 以下、荷物ごとに繰り返し ───
  → 送り状バーコードをスキャン
  → 該当する出荷指示が画面に表示される
  → 内容を確認（出荷先・箱数）
  → 「積込確認」ボタン
  ─── 繰り返し終了 ───
  → 全荷物の積込完了 → 「発送完了」ボタン

  ── B) トラック単位一括方式 ──
  → トラック番号をスキャン or 手入力
  → そのトラックに割り当てられた出荷指示一覧が表示される
  → 一覧を目視確認
  → 「一括発送完了」ボタン
```

**HTでの操作の詳細:**

| 操作 | 入力方法 | 補足 |
|------|---------|------|
| トラック番号 | 手入力 or リスト選択 | 配車システムから自動連携されている場合はリスト選択 |
| 送り状番号 | バーコードスキャン | 配送業者の伝票に印字されたバーコード |
| 出荷先確認 | 画面表示を目視 | スキャン結果に紐づく出荷先が自動表示される |
| 積込確認 | ボタン押下 | 1荷物ごと、またはトラック単位で確定 |
| 発送完了 | ボタン押下 | 全荷物積込後に最終確定 |

**発送完了時にシステムで起きること:**

```
「発送完了」ボタン押下
  → WMS: 在庫を引き落とし（引当済み → 出庫済み）
  → WMS: 出荷実績レコードを作成（タイムスタンプ + 作業者 + トラック番号）
  → ERP連携: 出荷実績データを送信（売上計上のトリガーになる）
  → 配送連携: 追跡番号を出荷先に通知（メール/SMS）※EC系の場合
  → HT画面: 「発送完了しました」トースト表示
  → 次の指示へ or メニューに戻る
```

**運用バリエーション:**

| パターン | 説明 | よくある現場 |
|---------|------|------------|
| 積込時スキャン | 1荷物ずつ送り状スキャンして積込確認 | 多品種・多出荷先の倉庫 |
| トラック一括確定 | トラック単位で積込リストを確認して一括確定 | 定期便・ルート配送 |
| 検品完了=出荷完了 | 検品が終われば即出荷扱い（積込管理なし） | 小規模倉庫・店舗間移動 |
| ゲート通過スキャン | 出荷ゲートを通過する際に自動読み取り | 大規模DC（固定スキャナ利用） |

---

### 出荷フロー全体のHT画面対応

| ステップ | HT画面 | 主な操作 | 省略可否 |
|---------|--------|---------|---------|
| 出荷指示受信 | 指示一覧画面 | リスト選択 or スキャン | 自動割当なら省略 |
| 現品登録 | ピッキング+検品画面 | ロケーション・商品スキャン+照合 | **必須** |
| 発送準備登録 | 梱包画面 | 箱番号・送り状スキャン | 小規模なら省略可 |
| 発送完了登録 | 発送完了画面 | 送り状スキャン or 一括確定 | **必須**（在庫引落トリガー） |

## 5. 入力方法の使い分け

| 入力対象 | 入力方法 | 理由 |
|---------|---------|------|
| 伝票番号・指示番号 | バーコードスキャン or リスト選択 | 伝票にバーコードが印刷されている場合はスキャン |
| ロケーション（棚番） | バーコードスキャン | 棚にバーコードラベルが貼付されている |
| 商品コード | バーコードスキャン | 商品にバーコードが印刷されている |
| 数量 | テンキー手入力 | 人が数える必要がある |
| ロット番号 | バーコードスキャン or 手入力 | ロットラベルがある場合はスキャン |
| 備考・コメント | テキスト手入力 | 自由入力が必要 |

### バーコードの種類

| 種類 | 用途 | 桁数例 |
|------|------|--------|
| JANコード（EAN） | 商品識別 | 13桁 / 8桁 |
| ITFコード | ケース単位の識別 | 14桁 |
| CODE128 | 汎用（社内コード等） | 可変長 |
| CODE39 | 汎用（英数字） | 可変長 |
| QRコード | 複合情報（ロット+期限等） | 可変長 |
| GS1-128 | 物流標準（AI付き） | 可変長 |

### GS1-128（AI: Application Identifier）の例

物流で使われるGS1-128バーコードは、AI（アプリケーション識別子）で情報を区分する。

```
(01)04912345678901(10)LOT-A001(17)260401(30)100
  │                  │            │          │
  ├─ AI01: GTIN      ├─ AI10: ロット ├─ AI17: 期限  ├─ AI30: 数量
```

1本のバーコードに複数の情報が含まれるため、スキャン後に「分解」して各フィールドに振り分ける処理が必要。
本アプリの `parseScanCode()` ユーティリティやダイアログデモの「分解」機能はこのパターンを簡易的に再現している。

## 6. 画面設計への反映

### HT画面の設計原則

| 原則 | 詳細 |
|------|------|
| **1画面1操作** | 複数の業務を1画面に詰め込まない |
| **上から下へ** | 操作順にフィールドを配置（スキャン → 手入力 → 確定） |
| **フィードバック即時** | スキャン成功/失敗は音・色・振動で即座に通知 |
| **大きなボタン** | グローブ着用でも押せるサイズ |
| **最小限の入力** | スキャンで埋まるフィールドは自動入力、手入力は数量のみ |
| **エラー時は明確に** | 何が間違っているか、どうすれば良いか表示 |

### 本アプリの画面構成との対応

```
PageLayout（共通ヘッダー + 戻るボタン + メニュー）
  │
  ├── ScanInput      ← バーコードスキャンで値を入力
  ├── ScanInput      ← 複数フィールドは順番にスキャン
  ├── NumberInput    ← 数量は手入力
  ├── ResultCard     ← スキャン結果や在庫情報を表示
  ├── SubmitButton   ← 確定ボタン（ローディング対応）
  └── FeedbackToast  ← 成功/エラーのフィードバック
```

## 7. 入力補助画面パターン

バーコードに複合情報が含まれる場合、スキャン後に値を分解して各フィールドに振り分ける「入力補助画面」が必要になる。

本アプリでは3つの実装パターンをデモしている（`/samples/mockups/dialog-demo`）。

| パターン | 実装方法 | 適したケース |
|---------|---------|-------------|
| モーダルダイアログ | `IonModal` + `@emit` | 軽い入力補助（選択肢が少ない、入力項目が1〜2個） |
| フルスクリーンモーダル | `IonModal` 全画面 + `@emit` | 複雑な入力補助（多項目、プレビュー必要） |
| ページ遷移 | `router.push` + `composable` | 独立した画面として成立する入力補助 |

### 使い分けの判断基準

```
入力補助の複雑さは？
  ├─ 選択肢から1つ選ぶだけ → モーダル（or SelectPopup）
  ├─ スキャン+分解+確認 → フルスクリーンモーダル
  └─ 独立した画面として使い回したい → ページ遷移
```

### データの返し方

| パターン | 送り方 | 受け取り方 |
|---------|--------|-----------|
| モーダル系 | `emit('confirm', value)` | 親コンポーネントの `@confirm` ハンドラ |
| ページ遷移 | `useScanTransfer().send(value)` + `router.back()` | 親ページの `onActivated` で `receive()` |
