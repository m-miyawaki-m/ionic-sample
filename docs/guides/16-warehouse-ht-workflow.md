# 倉庫HT（ハンディターミナル）業務フロー

倉庫現場でハンディターミナル（HT）を利用する際の典型的な業務フローを整理する。
本アプリの画面設計はこのフローに基づいている。

## 1. 基本原則

倉庫HTの操作は **「スキャンできるものはスキャン、数量だけ手入力、最後に確定」** が基本。

| 原則 | 理由 |
|------|------|
| スキャンで入力できるものは手入力しない | 入力ミス防止、作業速度向上 |
| 数量は人が数えて手入力する | バーコードに数量情報はない |
| 確定は必ずボタン操作 | 誤登録防止のため明示的な操作が必要 |
| 1トランザクションは短く完結させる | 作業中断・端末障害時のデータロスを最小化 |

---

## 2. 共通フロー

すべての業務は以下の共通フローに従う。

```
メニュー画面
  └→ 業務画面を選択
       └→ 伝票/指示を特定（スキャン or リスト選択）
            └→ ロケーションをスキャン
                 └→ 商品バーコードをスキャン
                      └→ 数量を手入力（必要な場合）
                           └→ 確定ボタン押下
                                └→ サーバーに送信
                                     └→ 次の商品 or 完了
```

### ステップごとの詳細

| ステップ | 操作 | 補足 |
|---------|------|------|
| 業務選択 | メニュー画面からタップ | 作業者の権限で表示を制御する場合もある |
| 伝票/指示の特定 | 紙伝票のバーコードスキャン or 画面リストから選択 | 予め割り当てられた指示が表示されるケースもある |
| ロケーション入力 | 棚に貼付されたバーコードをスキャン | 棚番は `A-03-02` のような形式が多い |
| 商品入力 | 商品に印刷されたバーコードをスキャン | JANコード、ITFコード、社内コードなど |
| 数量入力 | テンキーで手入力 | +1カウントアップ方式の場合もある |
| 確定 | ボタン押下 | サーバーと通信、結果をフィードバック |

## 3. 業務別フロー

### 3.1 入荷検品

入荷した商品が伝票通りか確認し、在庫に登録する。

```
入荷検品画面を開く
  → 入荷伝票番号をスキャン（または手入力）
  → 伝票内容が画面に表示される
  ─── 以下、商品ごとに繰り返し ───
  → ロケーション（格納先）をスキャン
  → 商品バーコードをスキャン
  → 数量を手入力
  → 「登録」ボタン → サーバー送信
  ─── 繰り返し終了 ───
  → 全品登録完了 → 伝票完了処理
```

**対応画面:** `ReceivingPage.vue`

### 3.2 出荷検品

出荷指示に対して正しい商品が揃っているか照合する。

```
出荷検品画面を開く
  → 出荷指示番号をスキャン（またはリストから選択）
  → 指示内容が表示される（品目・数量・出荷先）
  ─── 以下、商品ごとに繰り返し ───
  → 商品バーコードをスキャン
  → 指示と照合（一致 → OK音、不一致 → エラー音 + 警告表示）
  → 数量を確認（必要なら手入力）
  ─── 繰り返し終了 ───
  → 全品照合完了 → 「確定」ボタン
```

**ポイント:** 入荷と異なり「照合」がメイン。スキャンした商品が指示にあるかどうかをリアルタイムで判定する。

**対応画面:** `ShippingPage.vue`

### 3.3 棚卸し

実際の在庫数をカウントし、システム在庫との差異を確認する。

```
棚卸し画面を開く
  → ロケーションをスキャン
  ─── 以下、商品ごとに繰り返し ───
  → 商品バーコードをスキャン
  → 実数量を手入力
  → 「登録」ボタン
  ─── 繰り返し終了 ───
  → ロケーション完了 → 次のロケーションへ
```

**ポイント:** 伝票がない。ロケーション単位で棚にある商品をすべてカウントする。システム在庫との差異は後工程で確認。

**対応画面:** `StocktakingPage.vue`

### 3.4 在庫照会

特定の商品やロケーションの在庫情報を確認する（登録は行わない）。

```
在庫照会画面を開く
  → 商品バーコードをスキャン（またはロケーションをスキャン）
  → 在庫情報が表示される（ロケーション、数量、ロット等）
```

**ポイント:** 参照のみ。データ登録がないため確定ボタンは不要。

**対応画面:** `InventoryPage.vue`

### 3.5 ロケーション移動

商品を別の棚に移す際に記録する。

```
ロケーション移動画面を開く
  → 移動元ロケーションをスキャン
  → 商品バーコードをスキャン
  → 数量を手入力
  → 移動先ロケーションをスキャン
  → 「登録」ボタン → サーバー送信
```

**ポイント:** 移動元と移動先の2つのロケーションが必要。

**対応画面:** `RelocationPage.vue`

## 4. 入力方法の使い分け

| 入力対象 | 入力方法 | 理由 |
|---------|---------|------|
| 伝票番号・指示番号 | バーコードスキャン or リスト選択 | 伝票にバーコードが印刷されている場合はスキャン |
| ロケーション（棚番） | バーコードスキャン | 棚にバーコードラベルが貼付されている |
| 商品コード | バーコードスキャン | 商品にバーコードが印刷されている |
| 数量 | テンキー手入力 | 人が数える必要がある |
| ロット番号 | バーコードスキャン or 手入力 | ロットラベルがある場合はスキャン |
| 備考・コメント | テキスト手入力 | 自由入力が必要 |

### バーコードの種類

| 種類 | 用途 | 桁数例 |
|------|------|--------|
| JANコード（EAN） | 商品識別 | 13桁 / 8桁 |
| ITFコード | ケース単位の識別 | 14桁 |
| CODE128 | 汎用（社内コード等） | 可変長 |
| CODE39 | 汎用（英数字） | 可変長 |
| QRコード | 複合情報（ロット+期限等） | 可変長 |
| GS1-128 | 物流標準（AI付き） | 可変長 |

### GS1-128（AI: Application Identifier）の例

物流で使われるGS1-128バーコードは、AI（アプリケーション識別子）で情報を区分する。

```
(01)04912345678901(10)LOT-A001(17)260401(30)100
  │                  │            │          │
  ├─ AI01: GTIN      ├─ AI10: ロット ├─ AI17: 期限  ├─ AI30: 数量
```

1本のバーコードに複数の情報が含まれるため、スキャン後に「分解」して各フィールドに振り分ける処理が必要。
本アプリの `parseScanCode()` ユーティリティやダイアログデモの「分解」機能はこのパターンを簡易的に再現している。

## 5. 画面設計への反映

### HT画面の設計原則

| 原則 | 詳細 |
|------|------|
| **1画面1操作** | 複数の業務を1画面に詰め込まない |
| **上から下へ** | 操作順にフィールドを配置（スキャン → 手入力 → 確定） |
| **フィードバック即時** | スキャン成功/失敗は音・色・振動で即座に通知 |
| **大きなボタン** | グローブ着用でも押せるサイズ |
| **最小限の入力** | スキャンで埋まるフィールドは自動入力、手入力は数量のみ |
| **エラー時は明確に** | 何が間違っているか、どうすれば良いか表示 |

### 本アプリの画面構成との対応

```
PageLayout（共通ヘッダー + 戻るボタン + メニュー）
  │
  ├── ScanInput      ← バーコードスキャンで値を入力
  ├── ScanInput      ← 複数フィールドは順番にスキャン
  ├── NumberInput    ← 数量は手入力
  ├── ResultCard     ← スキャン結果や在庫情報を表示
  ├── SubmitButton   ← 確定ボタン（ローディング対応）
  └── FeedbackToast  ← 成功/エラーのフィードバック
```

## 6. 入力補助画面パターン

バーコードに複合情報が含まれる場合、スキャン後に値を分解して各フィールドに振り分ける「入力補助画面」が必要になる。

本アプリでは3つの実装パターンをデモしている（`/samples/dialog-demo`）。

| パターン | 実装方法 | 適したケース |
|---------|---------|-------------|
| モーダルダイアログ | `IonModal` + `@emit` | 軽い入力補助（選択肢が少ない、入力項目が1〜2個） |
| フルスクリーンモーダル | `IonModal` 全画面 + `@emit` | 複雑な入力補助（多項目、プレビュー必要） |
| ページ遷移 | `router.push` + `composable` | 独立した画面として成立する入力補助 |

### 使い分けの判断基準

```
入力補助の複雑さは？
  ├─ 選択肢から1つ選ぶだけ → モーダル（or SelectPopup）
  ├─ スキャン+分解+確認 → フルスクリーンモーダル
  └─ 独立した画面として使い回したい → ページ遷移
```

### データの返し方

| パターン | 送り方 | 受け取り方 |
|---------|--------|-----------|
| モーダル系 | `emit('confirm', value)` | 親コンポーネントの `@confirm` ハンドラ |
| ページ遷移 | `useScanTransfer().send(value)` + `router.back()` | 親ページの `onActivated` で `receive()` |
