# HT 画面設計シート 拡張 設計書

## 1. 目的

[2026-04-07 Web→HT画面検討チェックリスト設計書](2026-04-07-web-ht-screen-checklist-design.md) に基づき作成した [画面設計シート テンプレート (`19-ht-screen-design-sheet-template.md`)](../../guides/19-ht-screen-design-sheet-template.md) について、**Web 概要設計を入力として HT 画面を設計する** という実際の使い道に対し、起点（最初の一手）と中身が薄いという問題を解消する。

具体的には：

- **起点（とっかかり）が不明瞭**: 現テンプレは空白フォームのみで、Web 概要設計のどこから手をつけるかが見えない
- **概念データマッピングが無い**: 現テンプレの「データ」セクションは「主エンティティ」「識別方式」しか持たず、Web の ER 図 / エンティティ定義を取り込む場所が無い
- **画面項目マッピング表が浅い**: 6列構成で「出所エンティティ」「型/桁」「利用種別（入力以外の役割）」「デフォルト値」が無い
- **画面項目利用種別の概念が無い**: 「入力方式」と「役割としての利用種別（入力/表示/自動/計算/隠し 等）」が分離されていない
- **1 Web 画面 → N HT 画面の分解構造が無い**: 1:N の対応関係を表現する場所がどこにも無い

## 2. スコープ

### 2.1 対象

- 既存ファイル `docs/guides/19-ht-screen-design-sheet-template.md` の拡張
- 新規ファイル `docs/guides/20-ht-screen-decomposition-sheet-template.md` の作成

### 2.2 対象外

- `docs/guides/18-ht-common-design-guideline.md` の変更（前回 spec で別途整備済み）
- `docs/superpowers/specs/2026-04-07-web-ht-screen-checklist-design.md` の変更
- 各画面に対する実シートの作成（本 spec の成果物を使う後続作業）
- Web 概要設計ドキュメント自体の作成・整備

## 3. 前提

### 3.1 Web 概要設計として揃っている資料

ユーザの確認により、以下4種が手元にある：

- 画面項目一覧（画面項目定義書）
- ドメインモデル / ER図 / エンティティ定義
- 画面遷移図 / 業務フロー
- 画面レイアウト / ワイヤーフレーム

これらすべてを取込元として参照できる形にする。

### 3.2 変換の粒度

**1 Web 画面 → 複数 HT 画面 (1:N)** が見込み。
このため「分解」の表現が必須となる。

### 3.3 分解の表現方針

「**親 Web 画面ごとに分解シートを別途1枚作る**」をユーザが選択。
個別 HT シートは分解シートの存在を前提とし、起点情報で親分解シートにリンクで戻れるようにする。

### 6.3 ◆ 概念データマッピング セクション（新設）

「◆ データ」セクションの **直前** に新セクションを追加。

```markdown
### ◆ 概念データマッピング

Web の概要設計（ER図 / エンティティ定義書）から、この画面が扱うエンティティを抽出して整理する。

| エンティティ | Web側の役割 | HT側の扱い | 取得タイミング | 識別子 |
|---|---|---|---|---|
|  |  |  |  |  |
|  |  |  |  |  |
```

#### 列の定義

| 列 | 説明 |
|---|---|
| エンティティ | ドメインモデル上の名前（伝票・商品・ロケ・コンテナ・作業者 等） |
| Web側の役割 | 業務ルート（画面の主たる対象）／マスタ／コンテキスト／業務オブジェクト 等 |
| HT側の扱い | 主軸／照合対象／補助情報／表示のみ／自動付与／省略 |
| 取得タイミング | 起動時／スキャン都度／確定時／オフラインキャッシュ |
| 識別子 | このエンティティを HT で識別する手段（バーコード／QR／RFID／コード値 等） |

**狙い**:
- 画面項目（フィールド）の前に **エンティティ単位の整理** を強制 → 画面項目マッピングがブレない
- Web の ER 図 / エンティティ定義書 を直接転記する場所になる → 「とっかかり」が物理的に存在する
- どのエンティティに識別子があるかが一目でわかる → ハード連携の検討が早くなる

```markdown
| Web項目名 | 出所エンティティ | 型/桁 | HT 利用種別 | HT 入力方式 | 必須/任意 | デフォルト | 備考 |

```

#### 列の意味

| 列 | 説明 | 既存/新規 |
|---|---|---|
| Web項目名 | Web の画面項目定義書 から転記 | 既存 |
| 出所エンティティ | この項目がどのエンティティ由来か（◆ 概念データマッピングと紐づく） | 新 |
| 型/桁 | データ型と桁数（Web の項目定義書から転記） | 新 |
| HT 利用種別 | この項目の役割（下記凡例から1つ選ぶ） | 新（最重要） |
| HT 入力方式 | `利用種別=入力` の場合のみ埋める（既存語彙） | 既存（リネーム） |
| 必須/任意 | HT 版での必須性（Web と異なってよい） | 既存 |
| デフォルト | 初期値・自動付与値（自動/計算 の場合は必須） | 新 |
| 備考 | 補足事項 | 既存 |

#### 例示行

```markdown
| (例) 出荷指示番号 | 出荷指示 | string/12 | 入力 | スキャン(バーコード) | 必須 | - | 手入力フォールバック有 |
| (例) 商品コード | 商品 | string/13 | 入力 | スキャン(バーコード) | 必須 | - | JAN対応 |
| (例) 商品名 | 商品 | string/40 | 参照 | - | - | - | スキャン後にマスタから取得して表示 |
| (例) ロケーション | ロケーション | string/8 | 入力 | スキャン(QR) | 必須 | - |  |
| (例) 数量 | - | number/4 | 入力 | テンキー | 必須 | 1 | 0/負数不可 |
| (例) 単価 | 商品 | number/8 | 隠し | - | - | - | API 送信のみ、画面非表示 |
| (例) 検品時刻 | - | datetime | 自動 | - | 必須 | now() | 確定押下時に自動付与 |
| (例) 作業者ID | 作業者 | string/8 | 自動 | - | 必須 | login.userId | ログイン情報から |
| (例) 担当者氏名 | 作業者 | string/40 | 表示 | - | - | - | 自動と紐づき表示 |

```

#### HT 利用種別の凡例（新ボキャブラリ）

```markdown
**HT 利用種別の語彙:**

- `入力` — ユーザーが手で入力する。HT 入力方式列も埋める
- `参照` — マスタ等から取得して画面に表示する（読み取り専用）
- `表示` — 別項目に紐づいて表示する（参照との区別: 出所が明示的でない／別経路で来る）
- `自動` — ログイン情報・現在時刻・端末ID・GPS・前画面の値等から自動付与
- `計算` — 他項目から算出（例: 合計 = 数量 × 単価）
- `隠し` — API 送信ペイロードに含むが画面に表示しない
- `省略` — Web には存在するが HT 版では使わない
```

#### 既存「入力方式」語彙の扱い

現状の「入力方式」語彙（スキャン3種・テンキー・選択リスト・ソフトキー・表示のみ・省略・自動）はそのまま `HT 入力方式` 列の値ボキャブラリとして残す。**`利用種別=入力` のときだけ埋める**。`表示のみ` `省略` `自動` は新しい「HT 利用種別」と概念が重なるが、入力方式列での表記としては互換性のため残す（凡例に注記）。

#### 利用種別 × 入力方式 の関係

| 利用種別 | 入力方式列 |
|---|---|
| 入力 | 必ず埋める |
| 参照 / 表示 / 自動 / 計算 / 隠し / 省略 | `-` を入れる |

### 6.5 「使い方」の書き換え

現状の4ステップの抽象的な「使い方」をワークフロー型に書き直す。

````markdown
## 使い方

### 全体の流れ

```
Web 概要設計 → 分解シート (20) → 個別 HT シート (19) → 実装
```

### 起点: ここから始める

1. **親 Web 画面の [分解シート (20)](20-ht-screen-decomposition-sheet-template.md) を先に作る**
   - 1 Web 画面 → 何枚の HT シートに分けるかを最初に決める
   - 分解シートが無いまま個別シートを書き始めない
2. 分解シートで「N 枚作る」と決まったら、その N 枚分だけ本ファイル (19) をコピー
3. コピー先: `ht-screen-design-sheets/<画面名>.md`
4. 下記「埋める順序」に従って書く

### 埋める順序（推奨）

| # | セクション | 何をするか |
|---|---|---|
| 1 | 起点情報 | 親分解シートへのリンク・兄弟シート・HT化区分（A/B/C）を記入 |
| 2 | ◆ 概念データマッピング | Web の ER 図 / エンティティ定義書 から、この画面が扱うエンティティを5列の表に転記 |
| 3 | 画面項目マッピング表 | Web の画面項目定義書 を1行ずつ取り出して8列の表に転記。**この時点で「利用種別」を決め切る** |
| 4 | ◆ 業務フロー | 親分解シートのフロー欄から、この HT 画面の担当部分だけを切り出す |
| 5 | ◆ データ | ◆ 概念データマッピングの結果を要約 |
| 6 | ◆ 入力 | マッピング表の `利用種別=入力` の項目を集約 |
| 7 | ◆ UI | Web のワイヤーフレーム を見て HT 用に調整 |
| 8 | ◆ 通信 / オフライン, ◆ ハード連携, ◆ エラー / 事後入力, ◆ 権限 | 上記をもとに具体化（[HT共通設計ガイドライン](18-ht-common-design-guideline.md) と同じならスキップ可） |
| 9 | ◆ HT共通ガイドからの差分 | 共通ガイドと違う決定があれば書く。なければ「なし」 |
````
