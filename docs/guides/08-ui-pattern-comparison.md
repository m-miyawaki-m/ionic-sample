# UIパターン比較ガイド

本ドキュメントでは、倉庫在庫管理アプリ（Ionic Vue + TypeScript + Capacitor）の開発過程で検討した各画面のUIパターンを比較・整理する。
開発中はハンバーガーメニューから各パターンを切り替えて比較できるようになっている。

## 1. メニュー表示パターン（3種）

ハンバーガーメニュー押下時にパターン切替リストをどう表示するかについて、3パターンを検討した。

### パターン一覧

| # | パターン | 概要 | 採用 |
|---|---------|------|------|
| A | **Popover** | ボタン付近にドロップダウン表示 | **採用** |
| B | ActionSheet | 画面下部からスライドアップ | - |
| C | サイドメニュー（Drawer/Modal） | 画面横からスライドイン | - |

### 比較表

| 観点 | A) Popover | B) ActionSheet | C) サイドメニュー |
|------|-----------|---------------|------------------|
| **表示位置** | ボタン直下 | 画面下部 | 画面左端または右端 |
| **表示アニメーション** | フェードイン（軽量） | 下からスライドアップ | 横からスライドイン |
| **情報量** | 少量向き（5〜10項目） | 中量向き（5〜15項目） | 大量向き（制限なし） |
| **操作ステップ** | 1タップで表示・選択 | 1タップで表示・選択 | 1タップで表示・選択 |
| **画面の占有度** | 小（ボタン周辺のみ） | 中（下半分を覆う） | 大（画面の30〜80%を覆う） |
| **直感性** | 高い（ボタンとの関連が明確） | 中（ボタンから離れた位置に表示） | 中（ナビゲーション用途なら自然） |
| **モバイル親和性** | 高い | 非常に高い（iOS標準パターン） | 高い（ただし大げさな印象になりやすい） |
| **実装コスト** | 低い | 低い | 中〜高（状態管理が必要） |

### メリット・デメリット

#### A) Popover（採用）

- **メリット**
  - 軽量で表示が速い
  - ボタン直下に表示されるため、どのボタンから開いたかが直感的にわかる
  - 画面の占有面積が小さく、背景コンテンツの文脈を失わない
  - 項目数が少ない場合に最適なUIサイズ感
- **デメリット**
  - 項目数が多いとスクロールが必要になり見づらい
  - 画面端に近いボタンから呼び出すと表示位置がずれる場合がある

#### B) ActionSheet

- **メリット**
  - iOS/Androidの標準パターンであり、ユーザーが操作に迷わない
  - 画面下部からの表示は片手操作と相性が良い
  - キャンセルボタンが標準で付くため、誤操作時のリカバリが容易
- **デメリット**
  - ボタンとの位置的関連性が薄い（どの操作のメニューか直感的にわかりにくい）
  - 「破壊的操作の確認」という印象が強く、単なるメニューには大げさ
  - 画面下半分を覆うため、背景情報が見えなくなる

#### C) サイドメニュー（Drawer/Modal）

- **メリット**
  - 大量のメニュー項目をスクロールで表示可能
  - アプリ全体のナビゲーションとして使う場合は標準的なパターン
  - 階層構造やグループ分けが可能
- **デメリット**
  - 少数の項目に対しては画面の占有面積が過大
  - 開閉の状態管理が必要で実装コストが上がる
  - 「パターン切替」程度の用途には重すぎる

### 採用理由（Popover）

本アプリのメニューは「UIパターンの切替」が主な用途であり、項目数は3〜5程度と少ない。この条件では:

1. **軽量さ**: Popoverはオーバーレイのレンダリングコストが最も低く、ユーザーの操作フローを妨げない
2. **直感性**: ハンバーガーメニューボタンの直下に表示されるため、「このボタンのメニュー」という関係が一目でわかる
3. **項目数との相性**: 5項目以下であればPopoverの表示領域に収まり、スクロールも不要
4. **実装のシンプルさ**: `IonPopover`コンポーネントの`event`プロパティにクリックイベントを渡すだけで位置が自動計算される

> **確認方法**: ComponentsPageサンプルにて、Popover / ActionSheet / サイドメニューの3パターンを実際に切り替えて比較できる。ハンバーガーメニューから「メニュー表示パターン」を選択すること。

## 2. ホーム画面レイアウト（3種）

ホーム画面（業務メニュー一覧）の表示形式として3パターンを検討した。

### パターン一覧

| # | パターン | 概要 |
|---|---------|------|
| A | **リスト型** | `IonList` + `IonItem`による縦並びリスト。情報量が多い場合に向く |
| B | **グリッド（カード）型** | `IonGrid` + `IonCard`による2〜3列配置。視認性が高い |
| C | **大ボタン型** | 大きなタップ領域を持つボタンを縦並び配置。手袋環境向け |

### 比較表

| 観点 | A) リスト型 | B) グリッド（カード）型 | C) 大ボタン型 |
|------|-----------|---------------------|-------------|
| **1画面の情報量** | 多い（8〜15項目表示可能） | 中（4〜9項目表示可能） | 少ない（3〜5項目表示可能） |
| **タップ領域サイズ** | 中（横幅いっぱい、高さ48px程度） | 大（カード全体がタップ領域） | 非常に大（高さ80〜120px） |
| **アイコンの視認性** | 低い（リスト左端に小アイコン） | 高い（カード中央に大アイコン） | 高い（ボタン内に大アイコン） |
| **説明文の表示** | 容易（サブタイトル行に表示） | 困難（スペースが限られる） | 困難（ボタン内はラベルのみ） |
| **手袋でのタップ** | やや困難（タップ領域が狭い） | 可能（カードが大きい） | 非常に容易（巨大なタップ領域） |
| **スクロール必要性** | 低い（多くの項目が1画面に収まる） | 中（項目数次第） | 高い（項目数5以上でスクロール必須） |
| **見た目の印象** | 業務的・堅実 | モダン・視覚的 | シンプル・大胆 |

### 各パターンの向いている場面

#### A) リスト型

- **推奨場面**: 各メニュー項目に説明文やバッジ（件数表示等）を付けたい場合
- **理由**: サブタイトル行にテキストを配置でき、右端にバッジやアイコンも追加しやすい。項目数が多くてもスクロールなしで一覧性を確保できる
- **具体例**: 「入荷登録（本日: 15件）」「出荷処理（未処理: 3件）」のように件数を右端に表示するケース

#### B) グリッド（カード）型

- **推奨場面**: 項目数が5以下で、各項目をアイコンで直感的に識別させたい場合
- **理由**: カード内にアイコンを大きく配置できるため、文字を読まなくても色と形で機能を判別できる。2列配置により画面の使用効率も良い
- **具体例**: 「入荷」「出荷」「棚卸」「検索」の4メニューをアイコンカードで並べるケース

#### C) 大ボタン型

- **推奨場面**: 手袋を着用した倉庫環境で使う場合、またはタッチ精度が低い端末を使用する場合
- **理由**: タップ領域が非常に大きい（高さ80〜120px、横幅100%）ため、手袋越しでも確実にタップできる。誤タップのリスクが最も低い
- **具体例**: 冷蔵倉庫で厚手の手袋をしたまま「入荷スキャン」「出荷スキャン」を操作するケース

### 推奨まとめ

| 条件 | 推奨パターン | 理由 |
|------|------------|------|
| 項目数5以下 | **B) グリッド（カード）型** | アイコンが大きく表示され、少ない項目数なら1画面に収まり視認性が最も高い |
| 手袋・倉庫環境 | **C) 大ボタン型** | タップ領域の大きさが最優先。手袋越しの誤タップを防止できるのはこのパターンのみ |
| 情報（説明文）を見せたい | **A) リスト型** | サブタイトルやバッジの表示領域が確保でき、業務に必要な付加情報を一覧で伝達可能 |

## 3. 業務画面フォームレイアウト（3種）

入荷登録・出荷処理などの業務画面におけるフォームレイアウトとして3パターンを検討した。

### パターン一覧

| # | パターン | 概要 |
|---|---------|------|
| A | **縦並びフォーム** | 全項目を縦1列に配置する標準的なフォーム |
| B | **グループ分け型（カード）** | `IonCard`でスキャン項目と手入力項目を視覚的に分離 |
| C | **ステッパー型** | 1項目ずつ表示し、完了後に次の項目へ進む |

### 比較表

| 観点 | A) 縦並びフォーム | B) グループ分け型 | C) ステッパー型 |
|------|-----------------|-----------------|---------------|
| **一覧性** | 高い（全項目が見える） | 中（グループ単位で見える） | 低い（1項目ずつ） |
| **入力速度** | 速い（Tab/次へで連続入力） | 速い（グループ内は連続入力） | 遅い（毎回「次へ」タップ） |
| **ミス防止** | 低い（見落としやすい） | 中（グループで注意喚起） | 高い（1項目ずつ確認） |
| **項目間の関連理解** | 中（位置関係で推測） | 高い（グループで意味付け） | 低い（前後の文脈が見えない） |
| **画面サイズ依存** | 中（長いとスクロール必要） | 中（カード分のスペース消費） | 低い（常に1項目のみ表示） |
| **フォーカス制御** | 不要 | 不要 | 必要（ステップの状態管理） |
| **実装コスト** | 低い | 中（カードとグルーピングのロジック） | 高い（ステップ管理、バリデーション制御） |

### 各パターンの詳細と推奨場面

#### A) 縦並びフォーム

- **特徴**: `IonList`内に`IonItem` + `IonInput`を縦に並べるシンプルな構成。フォームの標準パターン
- **推奨場面**:
  - **項目数が3以下の場合**: スクロール不要で全項目が一目で見え、最も効率的
  - **熟練ユーザーが使う場合**: 操作に慣れたユーザーは全項目を一度に把握して高速入力できるため、ステップ移動のオーバーヘッドが無いこのパターンが最速
- **理由**: 余計なUI要素がなく、入力→Tab→入力→送信の最短フローを実現できる。熟練者にとってステッパーの「次へ」ボタンは不要な操作ステップになる

#### B) グループ分け型（カード）

- **特徴**: スキャン項目（バーコード読取）を1つのカード、手入力項目（数量・備考等）を別のカードに分けて表示。カードのヘッダーに「スキャン」「手入力」等のラベルを付ける
- **推奨場面**:
  - **スキャンと手入力が混在する場合**: スキャン完了後に手入力フェーズに移ることが視覚的に明確
  - **入力モードが異なる項目が混在する場合**: 「カメラで読み取る項目」と「キーボードで入力する項目」が明確に分かれ、ユーザーが次に何をすべきか迷わない
- **理由**: 倉庫業務では「バーコードをスキャン→数量を手入力→登録」というフローが典型的。このフローを視覚的にカードで区切ることで、現在どのフェーズにいるかが一目でわかる。カード間の境界が「操作の切り替わり」を暗示するため、ユーザーのメンタルモデルと一致する

#### C) ステッパー型

- **特徴**: 画面上部にステップインジケーターを表示し、1項目（または1グループ）ずつ大きく表示。「次へ」「戻る」ボタンでステップを移動
- **推奨場面**:
  - **初心者ユーザー・ミス防止が最優先の場合**: 1項目ずつ入力を確認するため、入力漏れや誤入力のリスクが最も低い
  - **各項目にガイダンスが必要な場合**: 各ステップに説明文やヒントを大きく表示できるスペースがある
- **理由**: 初心者は多数の入力項目を一度に見ると圧倒され、どこから入力すべきか迷う。ステッパーは「今はこれだけ入力すればよい」という明確なガイドを提供する。また、各ステップでバリデーションを実行できるため、誤入力を即座にフィードバックできる

### 推奨まとめ

| 条件 | 推奨パターン | 理由 |
|------|------------|------|
| 項目数が3以下 | **A) 縦並びフォーム** | 全項目が1画面に収まり、追加のUI要素なしで最もシンプルかつ高速に入力完了できる |
| スキャンと手入力が混在 | **B) グループ分け型** | 操作モードの切り替わりを視覚的に示すことで、「次に何をすべきか」の判断コストを下げる |
| 初心者ユーザー・ミス防止 | **C) ステッパー型** | 1項目ずつ確認しながら進むため入力漏れが起きにくく、各ステップでのバリデーションも可能 |
| 熟練ユーザー | **A) 縦並びフォーム** | ステップ移動やグループ切替の操作が不要で、連続入力の最短フローを実現できる |

## 4. ローディング表示パターン（2種）

非同期処理（API通信等）中のユーザーへのフィードバック方法として2パターンを検討した。

### パターン一覧

| # | パターン | 概要 |
|---|---------|------|
| A | **全画面オーバーレイ（IonLoading）** | 半透明のオーバーレイとスピナーで画面全体を覆い、操作をブロック |
| B | **ボタン内スピナー** | 送信ボタン内にスピナーを表示し、ボタンのみ無効化。画面の他の部分は操作可能 |

### 比較表

| 観点 | A) 全画面オーバーレイ | B) ボタン内スピナー |
|------|-------------------|------------------|
| **操作ブロック** | 全画面（全操作不可） | ボタンのみ（他は操作可能） |
| **二重送信防止** | 確実（画面全体が無効化） | ボタンのみ（他の送信経路がある場合は不十分） |
| **ユーザーの印象** | 「待たされている」感が強い | 軽い・スムーズ |
| **キャンセル操作** | 不可（オーバーレイ解除まで） | 可能（戻るボタン等が押せる） |
| **処理時間の体感** | 長く感じやすい | 短く感じやすい |
| **実装コスト** | 低い（`IonLoading`をそのまま使用） | 中（ボタンの状態管理が必要） |
| **エラー時の復帰** | オーバーレイ解除後に再操作 | そのまま再入力・再送信可能 |

### 推奨

| 処理の種類 | 推奨パターン | 理由 |
|-----------|------------|------|
| **データ登録・更新** | **A) 全画面オーバーレイ** | 登録・更新処理は二重送信が致命的（在庫数の二重加算等）。画面全体をブロックすることで、処理完了まで他の操作を確実に防止できる。「待たされている」感はあるが、データの整合性を守ることが最優先 |
| **検索・照会** | **B) ボタン内スピナー** | 検索は二重実行しても副作用がないため、厳密なブロックは不要。検索条件の修正やキャンセル操作を許容することで、ユーザーは待ち時間中に次の操作を準備できる。体感的にも軽快な印象を与える |

### 補足: 処理時間による使い分け

- **0.5秒未満**: ローディング表示なし（一瞬で終わるため表示するとチラつく）
- **0.5秒〜3秒**: ボタン内スピナー（短い待ち時間に全画面ブロックは大げさ）
- **3秒以上**: 全画面オーバーレイ + メッセージ表示（「登録中です...」等。長時間ブロックする場合は理由を伝える）

## 5. スキャンフィードバックパターン（4種）

バーコードスキャン成功時のフィードバック方法として4パターンを検討した。

### パターン一覧

| # | パターン | 概要 |
|---|---------|------|
| A | **なし** | フィードバックなし。画面上の値変化のみ |
| B | **バイブのみ** | Capacitor Hapticsによる振動フィードバック |
| C | **トーストのみ** | `IonToast`による画面上部/下部の一時的なメッセージ表示 |
| D | **バイブ + トースト** | 振動と視覚的メッセージの両方 |

### 比較表

| 観点 | A) なし | B) バイブのみ | C) トーストのみ | D) バイブ+トースト |
|------|--------|-------------|---------------|-----------------|
| **認知チャネル** | なし | 触覚のみ | 視覚のみ | 触覚 + 視覚 |
| **騒音環境での認知** | 不可（見落とし多発） | 高い（振動は騒音に影響されない） | 中（画面を見ていれば認知可能） | 非常に高い |
| **画面非注視時の認知** | 不可 | 高い（手に伝わる） | 不可（画面を見ていない） | 高い |
| **情報量** | なし | 低い（成功/失敗の二値） | 高い（スキャン結果のテキスト表示可能） | 高い |
| **連続スキャン時** | 認知困難 | 快適（リズム感がある） | やや煩わしい（トーストが重なる） | 快適（振動でリズム、トーストで確認） |
| **バッテリー消費** | なし | 微量 | なし | 微量 |
| **実装コスト** | なし | 低い | 低い | 低い（AとBの組合せ） |

### 各パターンの適用場面

#### A) なし

- **適用場面**: 基本的に非推奨。開発初期のプロトタイプや、フィードバックが別の形で明示される場合（例: スキャン後に即座に画面遷移する場合）のみ
- **問題点**: 倉庫環境は騒がしく、ユーザーは常に画面を注視しているわけではない。フィードバックがないと「スキャンできたのかどうか」が不明確になり、同じバーコードを何度もスキャンしてしまう

#### B) バイブのみ

- **適用場面**: 連続スキャンが主な操作で、スキャン結果の詳細確認が不要な場合（例: 入荷検品で商品を次々にスキャンしていくフロー）
- **利点**: 触覚フィードバックは騒音環境でも確実に伝わり、画面を見ていなくても認知可能。連続スキャン時のリズム感が作業効率を上げる
- **制約**: 「何をスキャンしたか」の情報は伝えられないため、画面上の表示更新と併用が前提

#### C) トーストのみ

- **適用場面**: デバイスの振動機能が利用できない環境（Web版やバイブ非対応端末）、またはスキャン結果のテキスト確認が重要な場合
- **利点**: スキャンした商品名やコードをテキストで表示できるため、誤スキャン（隣の商品をスキャンしてしまった等）に気づきやすい
- **制約**: 画面を見ていないと認知できない。連続スキャン時にトーストが重なると煩雑

#### D) バイブ + トースト（推奨）

- **適用場面**: 本番運用のほぼ全てのケース
- **利点**: 触覚と視覚の2チャネルで冗長的にフィードバックするため、認知漏れのリスクが最も低い

### 推奨: D（バイブ + トースト）

倉庫環境での使用を前提とした場合、**D（バイブ + トースト）** を強く推奨する。理由は以下の通り:

1. **騒音環境への対応**: 倉庫ではフォークリフトの走行音、荷物の搬送音などが常に発生しており、音声フィードバック（ビープ音等）は聞こえない場合が多い。バイブ（触覚）はこの問題を完全に回避できる
2. **視線移動の最小化**: 作業者は商品ラベルとスキャナー（端末）の間で頻繁に視線を移動する。バイブにより「画面を見なくてもスキャン成功がわかる」状態を作ることで、視線移動の回数を減らし作業効率を向上させる
3. **誤スキャン検知**: トーストでスキャン結果（商品名やコードの一部）を表示することで、「隣の商品をスキャンしてしまった」「同じ商品を二度スキャンしてしまった」等のミスに気づく機会を提供する
4. **冗長性による確実性**: 単一チャネルのフィードバックは見落とし・感じ落としのリスクがあるが、2チャネルの組合せにより認知の確実性が大幅に向上する

### 実装時の注意点

- バイブの長さは**50〜100ms**が推奨。短すぎると感じ取れず、長すぎると連続スキャン時に不快
- トーストの表示時間は**1.5〜2秒**が推奨。短すぎると読めず、長すぎると次のトーストと重なる
- エラー時はバイブのパターンを変える（例: 短い振動2回）ことで、画面を見なくても成功/失敗を区別可能にする
- トーストの色を成功（緑系）/失敗（赤系）で分けることで、視覚的にも即座に判別できるようにする

## 6. 総合推奨（本番向け）

各カテゴリの推奨パターンを以下にまとめる。

### 推奨構成一覧

| カテゴリ | 推奨パターン | 主な理由 |
|---------|------------|---------|
| **メニュー表示** | A) Popover | 項目数が少なく（5以下）、ボタン直下の表示が直感的。軽量で実装もシンプル |
| **ホーム画面** | B) グリッド型 または C) 大ボタン型 | 倉庫環境では大ボタン型の操作性が優れる。一般環境ではグリッド型の視認性が最適 |
| **フォームレイアウト** | B) グループ分け型 | 倉庫業務はスキャンと手入力の混在が典型的であり、操作モードの切り替わりを視覚化することでミスを低減 |
| **ローディング表示** | 処理内容で使い分け | 登録・更新は全画面オーバーレイ（二重送信防止）、検索・照会はボタン内スピナー（軽快な操作感） |
| **スキャンフィードバック** | D) バイブ + トースト | 騒音環境でも確実に認知でき、触覚と視覚の2チャネルにより認知漏れリスクを最小化 |

### 環境別の推奨バリエーション

#### 冷蔵倉庫（手袋着用）

| カテゴリ | 推奨 | 補足 |
