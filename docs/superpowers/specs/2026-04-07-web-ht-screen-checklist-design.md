# Web→HT 画面検討チェックリスト 設計書

## 1. 目的

すでに Web システムが存在する／作成中の業務システムに対し、HT（ハンディターミナル）対応の Android アプリを追加する際、各画面について **何を検討すべきか** を漏れなく整理するためのチェックリストとその使い方を定める。

本書はプロジェクト固有の設計書ではなく、**Web → HT 化を行うあらゆる業務システムで再利用可能** な検討フレームとして作成する。倉庫管理を主想定としているが、他業種の現場業務系（製造・物流・店舗等）でも応用できる。

## 2. スコープと前提

### 2.1 想定する HT 端末

- Android OS ベースの業務用 HT（例: Xnavis 等）
- ハードウェアスキャナ（バーコード／QR）搭載
- 必要に応じて RFID リーダ搭載可能性あり
- 物理ボタン（スキャントリガ、ファンクションキー）あり
- 画面サイズは小さめ（10cm以下）、縦持ち固定が一般的
- 利用環境: 立ち作業／移動中／手袋／低照度／低温倉庫など

### 2.2 想定する Web⇄HT の関係

- Web 版が先行 or 並行で存在
- HT は Web の機能を **すべて移植するわけではない**（業務特性で取捨選択）
- HT 独自の追加機能（事後入力／一括登録／現場用途）が必要になる場合がある
- Web/HT の対応関係はケースバイケースで判断する

### 2.3 識別子の物理付与方式

- バーコード（1D）／QR（2D）／RFID の併存を前提とする
- 画面ごと・エンティティごとに方式が異なりうる

## 3. 全体方針: 3層モデル

チェックリスト全項目（100超）を 1画面ごとに全部チェックするのは非現実的。実運用では以下の3層に分けて回す。

### Layer 1: HT共通設計ガイドライン（一度だけ決める）

ほとんどの検討項目は画面ごとに違わないため、プロジェクト単位で1回決めて全画面に適用する。
別文書「**HT共通設計ガイドライン**」にまとめ、全画面はそれを継承する前提とする。

| カテゴリ | 共通で決める例 |
|---|---|
| ハード連携 | スキャナSDK選定、リーダ起動／停止規約、音／振動パターン、物理ボタン規約 |
| UI共通 | フォントサイズ最低基準、タップターゲット、色規約、状態バッジの位置、エラー表示位置 |
| 通信 | タイムアウト値、再送ポリシー、送信キュー、API認証方式、ローカルDB方式 |
| 運用／権限 | ログイン方式、ロール設計、操作ログ仕様、端末割当規則、配布方式 |
| エラー処理 | エラー画面テンプレ、二重押下防止規約、競合検出規約 |

チェックリストの「★共通」マーク付き項目は、画面ごとには再確認しない（差分があるときだけ画面設計シートに書く）。

### Layer 2: 画面ごとの STEP 0 判定（必須）

全画面で必ず行う。判定結果（A/B/C/D）が以降の検討の濃さを決める。

```
A. フルHT化      → STEP 1〜8 全部見る（★共通は流す）
B. 参照のみHT化   → STEP 1, 2, 4, 6, 8 を軽く見る
C. リカバリ専用   → STEP 1, 2, 4, 5, 7, 8（特に7を厚く）
D. HT化しない     → ここで終了
```

### Layer 3: 画面ごとの「画面設計シート」（A4 1枚）

A/B/C 判定された画面ごとに 1枚作成する。
チェックリスト全項目を埋めるのではなく、**この画面固有の決定事項** だけを書く。共通決定との差分を明確にするのが目的。

シートのテンプレは [9. 画面設計シート テンプレート](#9-画面設計シート-テンプレート) を参照。

### 3層の関係

| 成果物 | 役割 | 更新頻度 |
|---|---|---|
| HT共通設計ガイドライン | 画面横断の決定事項を集約 | 立ち上げ時に作成、以後は逸脱発見時に追記 |
| チェックリスト（本書） | 漏れ防止のリファレンス | 更新は稀 |
| 画面設計シート | 画面ごとの固有決定 | 画面追加・変更ごと |

## 4. STEP 構成

| STEP | カテゴリ | 主な問い |
|---|---|---|
| 0 | HT化判断 | この画面はHT化すべきか／参照のみ／リカバリ専用／不要か |
| 1 | 業務フロー | トリガ・完了条件・連続作業・中断再開・正常フロー以外の入り口 |
| 2 | データ観点 | 概念データ・I/O・画面項目・識別子・コードの情報量 |
| 3 | 入力 | スキャン駆動・方式の使い分け・手入力・物理ボタン・フォールバック |
| 4 | UI／レイアウト | 1画面1タスク・視認性・タップ領域・状態表示・片手操作 |
| 5 | 通信／オフライン | 即時/バッチ・再送・キャッシュ・競合解決 |
| 6 | ハード連携 | リーダSDK・電力・音/振動・物理ボタン・センサ |
| 7 | エラー／状態 | 通常エラー・競合・中断再開・事後入力（リカバリ入力） |
| 8 | 運用／権限 | 画面/アクション権限・ログ・承認フロー・運用規約 |

## 5. STEP 0「HT化判断」

**目的**: その画面をHT化すべきか／参照のみか／HT化しないかを最初に決める。これがNOなら以降のSTEPは不要。

### 5.1 判断のための問い

| # | 問い | 判断のヒント |
|---|---|---|
| 0-1 | この画面の主目的は何か？ | 参照／登録／更新／集計／マスタ設定／レポート |
| 0-2 | 現場作業中に開く必然性があるか？ | 「机上で開けば足りる」ならHT化対象外の有力候補 |
| 0-3 | 1セッションの入力が短時間で完結するか？ | HTは長時間の入力作業に向かない（数十秒〜数分が理想） |
| 0-4 | スキャン入力の機会があるか？ | スキャンが無いHT画面は基本的に旨味が薄い |
| 0-5 | この画面で扱うエンティティに識別子（バーコード/QR/RFID）が付いているか／付ける運用が現実的か？ | 付与済みなら効果大、未付与なら付与運用とセットで議論 |
| 0-6 | RFIDが付いているなら一括読み取り型UIに作り替える価値はあるか？ | 棚卸し等は桁違いの工数差 |
| 0-7 | 利用頻度はどの程度か？ | 月数回ならWebで十分なケースが多い |
| 0-8 | 利用者の作業姿勢・環境にHTが適合するか？ | 立ち作業・移動中・手袋・低照度・低温倉庫 |
| 0-9 | 紙運用やWebに比べて便益が明確か？ | 工数削減・誤入力削減・リアルタイム性のうちどれが効くか |
| 0-10 | 事後入力経路だけ用意すれば足りる画面ではないか？ | 通常はWeb、現場の例外時のみHT、というパターン |

### 5.2 判定アウトカム

| 区分 | 意味 | 以降のSTEP |
|---|---|---|
| **A. フルHT化** | 通常運用でHTから入力・参照・登録までできる | STEP 1〜8 全部 |
| **B. 参照のみHT化** | 現場で見るだけ。登録はWeb | STEP 1, 2, 4, 6, 8 |
| **C. リカバリ専用HT化** | 通常はWeb、事後入力／緊急時のみHT | STEP 1, 2, 4, 5, 7, 8（7重点） |
| **D. HT化しない** | Web専用 | 終了 |

## 6. STEP 1〜8 チェック項目

### STEP 1「業務フロー観点」

**目的**: HT版がどの業務フローのどの位置に座るかを定め、開始/中断/再開/終了の境目を明確にする。

| # | 問い | 判断のヒント |
|---|---|---|
| 1-1 | この画面は全体業務フローのどの位置か？ 前後の画面は何か？ | 例: 出荷指示一覧 → 現品登録 → 発送準備 |
| 1-2 | 画面を開くトリガは何か？ | メニュータップ／伝票スキャン／自動割当／前画面からの自動遷移 |
| 1-3 | 完了条件と確定操作は何か？ | 「全件スキャン後 確定ボタン」「明細1件ごとに即時送信」 |
| 1-4 | 1トランザクションは短時間で完結できる単位に切れているか？ | 長すぎるなら分割を検討 |
| 1-5 | 連続作業の流れは？ | 確定後に次の伝票/次のロケへ自動遷移するか／メニューに戻るか |
| 1-6 | 中断が起きうるシーンは？ | 電話・呼び出し・スリープ・端末持ち替え |
| 1-7 | 再開の境目はどこか？ | 「最後にサーバへ届いた件数」「最後にスキャンした商品」をどう示すか |
| 1-8 | 1人作業 or 協調作業か？ | 同一伝票を複数HTが触るなら排他制御要 |
| 1-9 | 正常フロー以外の入り口（事後入力モード）は要るか？ | STEP 7D（事後入力経路）と連動 |
| 1-10 | 業務上の例外（破損・品違い・欠品・返品）はこの画面で扱うか／別画面か？ | 扱うなら理由コード等の追加項目が要る |

### STEP 2「データ観点」

**目的**: Web画面の項目を1つずつ取り出し、HT版での扱いを決める。Web→HTで一番効く論点。

#### 2A. 概念データ（ドメインエンティティ）

| # | 問い |
|---|---|
| 2A-1 | この画面が扱う主たるエンティティは何か？（伝票・在庫・商品・ロケ・コンテナ・パレット・作業者 等） |
| 2A-2 | HT専用エンティティは要るか？（端末割当・作業セッション・スキャン履歴） |
| 2A-3 | Web版とエンティティの参照範囲は同じか／HTは絞るか？（例: 在庫はその端末担当の倉庫のみ） |
| 2A-4 | マスタの鮮度要件は？（リアルタイム必須／日次同期で十分／オフラインキャッシュ許容） |
| 2A-5 | マスタ件数はHTのストレージ・メモリで扱える量か？（数万件超ならサーバ問い合わせ型に） |

#### 2B. I/Oデータ

| # | 問い |
|---|---|
| 2B-1 | 入力: Web版で投入する項目のうち、HTで本当に必要なのはどれか？ |
| 2B-2 | HT専用に追加で取りたい項目は何か？（端末ID／作業者ID／タイムスタンプ／GPS／写真／音声メモ） |
| 2B-3 | 出力: 画面表示／APIペイロード／ローカル保存 のそれぞれで持つべき項目は何か？ |
| 2B-4 | API は Web と共用 or HT専用か？（軽量化のためHT専用エンドポイントを切るか） |
| 2B-5 | 1回のサーバ応答で返るデータ量は妥当か？（ページング・遅延読み込みの要否） |

#### 2C. 画面項目（フィールド種別マッピング）

Web側の項目を1つずつ取り出して以下の表を埋める作業を、この画面の必須成果物にする。

| Web項目 | HTでの扱い | 入力方式 | 必須/任意 | 備考 |
|---|---|---|---|---|
| 例: 商品コード | あり | スキャン(バーコード) | 必須 | 手入力フォールバック有 |
| 例: 数量 | あり | テンキー / +1ボタン | 必須 | 0/負数チェック |
| 例: 担当者氏名 | 表示のみ | （ログインから自動） | - | 入力不要 |
| 例: 備考（長文） | あり(短縮) | ソフトキー | 任意 | 全角制限 |
| 例: 添付ファイル | 省略 | - | - | Webのみ |

| # | 問い |
|---|---|
| 2C-1 | 各Web項目を `スキャン / テンキー / 選択リスト / 表示のみ / 省略` のどれにマップするか？ |
| 2C-2 | Web版での必須/任意を見直す余地はあるか？（HTでは省略してデフォルト埋めできるか） |
| 2C-3 | HT特有の追加表示項目は何か？（残数・前回作業者・期限近商品の警告・推奨ロケ） |
| 2C-4 | 桁数・文字種制限は？（HTでは半角強制／文字数を絞る） |

#### 2D. 識別子の物理付与方式

| # | 問い |
|---|---|
| 2D-1 | この画面で扱う各エンティティは バーコード(1D) / QR(2D) / RFID のどれで識別されるか？ |
| 2D-2 | 複数方式が共存するか？ 優先順位は？ |
| 2D-3 | 未付与のエンティティに付与運用は可能か？（コスト・現場負担） |

#### 2E. コードに含む情報量

| # | 問い |
|---|---|
| 2E-1 | スキャン値はID参照のみか、それとも複合フィールド（QRに伝票No+ロット+期限など）を埋め込んでいるか？ |
| 2E-2 | パース仕様（区切り文字／プレフィックス）は明文化されているか？ |
| 2E-3 | パース失敗時のエラーハンドリングはどうするか？ |

### STEP 3「入力観点」

**目的**: スキャン駆動を最大化し、手入力を最小化する。各項目の入力方式を明示する。

| # | 問い | 判断のヒント |
|---|---|---|
| 3-1 | スキャン化できる項目を最大化したか？ | 文字を1つでも手入力させていないか見直す |
| 3-2 | スキャン方式の使い分けは明確か？ | バーコード(1D) / QR(2D) / RFID をどの項目で使うか |
| 3-3 | スキャンは単発／連続／一括のどれか？ | 連続スキャン中は確定操作の扱いに注意 |
| 3-4 | 方式特性に依存するUIは意識したか？ | バーコード/QR=縦フロー、RFID=一括取得→差分表示の横フロー |
| 3-5 | スキャン後の自動処理はあるか？ | 自動カーソル移動／自動確定／自動次画面遷移 |
| 3-6 | 手入力が残る項目（数量等）の入力方式は？ | テンキー強制／+1ボタン／ステッパー／プリセット |
| 3-7 | IME回避は徹底したか？ | 半角強制／テンキー強制／ソフトキーボード抑止 |
| 3-8 | 物理ボタンの割当は？ ★共通 | スキャントリガ／Enter／戻る／F1-F4等 |
| 3-9 | スキャンが読めない場合のフォールバックは？ | 手入力経路／カメラスキャン／帳票照合 |
| 3-10 | 誤読・重複対策は？ | 同一タグ連続読み込み抑止／類似コード警告／RFIDの読取距離調整 |
| 3-11 | 入力値のバリデーションは即時か遅延か？ | スキャン直後にチェック → 不一致なら音/振動で警告 |
| 3-12 | 不一致時のリカバリ動線は？ | 「指示にない商品です」→ 強制登録 or キャンセル |

### STEP 4「UI／レイアウト観点」

**目的**: 小画面・手袋・低照度・移動中という現場条件に最適化する。

| # | 問い | 判断のヒント |
|---|---|---|
| 4-1 | 1画面1タスクになっているか？ | 詰め込みすぎていないか／ステップ分割の余地 |
| 4-2 | 重要情報の固定配置は？ | 伝票番号・進捗（3/10件）・状態バッジは常に上部 |
| 4-3 | フォントサイズの最低基準を満たすか？ ★共通 | 倉庫内の暗所・遠目で見える大きさ |
| 4-4 | タップターゲットの最低サイズは？ ★共通 | 手袋でも押せる大きさ（推奨44px以上） |
| 4-5 | 色覚対応・コントラストは十分か？ ★共通 | OK/NGを色だけに頼らない（音・形・文言併用） |
| 4-6 | 画面の向きは？ ★共通 | 縦持ち固定 or 回転対応 |
| 4-7 | 即時フィードバックは備わっているか？ ★共通 | スキャン成功音／失敗音／振動／LED／色変化 |
| 4-8 | 状態バッジの表示は？ | オフライン中／事後入力モード／通信中／弱電波 |
| 4-9 | 戻る操作の安全性は？ | 途中入力の喪失を防ぐ確認ダイアログ／一時保存 |
| 4-10 | ローディング状態は明確か？ ★共通 | 通信中はボタン無効化＋スピナ／二重押下防止と連動 |
| 4-11 | エラー表示の位置は適切か？ ★共通 | スクロール不要な位置／フォーカスを当てる |
| 4-12 | 画面遷移はミニマムか？ | 1業務で開く画面数を減らす／パンくず代替の進捗表示 |
| 4-13 | 片手操作を想定しているか？ ★共通 | 主要ボタンを下部・親指届く位置に配置 |
| 4-14 | 数値表示の桁・単位は誤読しにくいか？ | 大きな桁の区切り／単位を必ず併記 |

### STEP 5「通信／オフライン観点」

**目的**: HTは電波が弱い・切れる前提。送信タイミング・再送・整合性を画面ごとに決める。

| # | 問い | 判断のヒント |
|---|---|---|
| 5-1 | この画面はオンライン必須か、オフライン許容か？ | 必須なら弱電波時の挙動も決める |
| 5-2 | 送信は即時送信かバッチ送信か？ | 1スキャンごと vs 確定ボタンでまとめて |
| 5-3 | タイムアウト値は何秒か？ 超過時の挙動は？ ★共通 | 再試行／キャンセル／キュー投入 |
| 5-4 | 弱電波時のユーザ通知は？ ★共通 | 状態バッジ／音／自動再試行中の表示 |
| 5-5 | 送信キューを持つか？ 順序保証は要るか？ | 順序が業務上重要な画面（在庫更新等）は要設計 |
| 5-6 | 再送ポリシーは？ ★共通 | 自動再送回数／指数バックオフ／手動再送ボタン |
| 5-7 | ローカル保存の範囲は？ | 一時（メモリ）／永続（SQLite/IndexedDB）／暗号化要否 |
| 5-8 | マスタデータのキャッシュは？ | 起動時同期／日次同期／オンデマンド／TTL |
| 5-9 | キャッシュの古さ許容はどこまで？ | 「3時間以内なら使う」等の基準 |
| 5-10 | 競合解決ポリシーは？ | 楽観ロック（バージョン番号）／悲観ロック（事前ロック取得） |
| 5-11 | 同一画面を複数HTで開いた時の挙動は？ | 排他取得／警告表示／後勝ち |
| 5-12 | オフライン中に登録したデータの扱いは？ | サーバ反映時に競合検出する仕組み／ID採番（UUID/サーバ採番） |
| 5-13 | バックグラウンド同期の可否 | アプリがバックグラウンドでもキューを送るか／OS制約に注意 |
| 5-14 | ペイロードサイズは妥当か？ | 写真・添付があるなら圧縮／分割送信 |

### STEP 6「ハード連携観点」

**目的**: HT固有のハードウェア（スキャナ／RFID／カメラ／音／振動／プリンタ等）の使用可否と起動・停止を整理する。

| # | 問い | 判断のヒント |
|---|---|---|
| 6-1 | この画面で使うリーダ系統は何か？ | バーコード/QR用SDK／RFID用SDK／カメラスキャン |
| 6-2 | リーダSDKの起動／停止タイミングは？ ★共通 | onMounted / onBeforeUnmount で確実にopen/close |
| 6-3 | リーダのclose漏れを防ぐ仕組みは？ ★共通 | 画面遷移・例外時のクリーンアップ／グローバルguard |

## 7. ★共通マークの読み方

各STEPの問いのうち末尾に **★共通** を付けた項目は、Layer 1「HT共通設計ガイドライン」で1度だけ決めて全画面に適用するものを示す。
画面ごとには **共通決定からの差分が出る場合のみ** 画面設計シートに記載する。

## 8. 使い方サマリ

```
新しい画面の検討フロー
━━━━━━━━━━━━━━━━━━━━━━

1. STEP 0 で A/B/C/D を判定
        │
   ┌────┴─────┬─────┬─────┐
   ↓          ↓     ↓     ↓
   A          B     C     D
   │          │     │     │
   ↓          ↓     ↓    終了
2. 適用STEPを開く（Aは全部、B/Cは絞られる）

3. 各STEPの項目を流し読み
   ・★共通 の項目 → 共通ガイドの規約をそのまま適用（差分があれば記録）
   ・★共通でない項目 → 画面設計シートに決定事項を書く

4. 画面設計シート（A4 1枚）を完成させる
        │
        ↓
5. レビュー → 実装へ
```
